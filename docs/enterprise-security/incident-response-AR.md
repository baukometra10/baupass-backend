# خطة الاستجابة للحوادث الأمنية (IR)

## 1. التصنيف

| المستوى | الوصف | هدف الاستجابة |
|---------|--------|----------------|
| P1 | تسريب بيانات أو اختراق نشط | &lt; 1 ساعة إشعار داخلي |
| P2 | تعطيل خدمة واسع | &lt; 4 ساعات |
| P3 | حادث محدود (مستأجر واحد) | &lt; 24 ساعة |
| P4 | محاولة فاشلة / مسح | توثيق أسبوعي |

## 2. أدوار

| الدور | المسؤولية |
|-------|-----------|
| Incident Commander | تنسيق، قرارات، تواصل مع العميل |
| Technical Lead | عزل، logs، patch |
| Legal / DPO | إخطار جهة حماية البيانات إن لزم |
| Communications | رسالة للعملاء المتأثرين |

## 3. خطوات (ملخّص)

1. **اكتشاف** — تنبيه SIEM، عميل، أو فريق داخلي.
2. **احتواء** — تعطيل token/mفتاح API، حظر IP، إيقاف مستأجر متأثر.
3. **تحقيق** — جمع `access_logs`, `platform_events`, نسخ DB read-only.
4. **استئصال** — patch، تدوير أسرار، إعادة نشر.
5. **استعادة** — استعادة من backup إن لزم (`docs/enterprise-backup-restore-runbook.md`).
6. **دروس مستفادة** — تقرير post-mortem خلال 5 أيام عمل.

## 4. إخطار العملاء

- عقد Enterprise: بند إخطار خلال **72 ساعة** (أو حسب DPA).
- توثيق: ما الذي تسرّب، من المتأثر، الإجراءات المتخذة.
